## 2. Data We Collect

We collect personal data in the categories listed below.

### 2.1 Account and Authentication Data

When you register for the App, we collect the following information:

- **Email address** (required)
- **Password** — stored as an irreversibly hashed value via Supabase Authentication; never stored in plain text anywhere
- **Username** (unique, at least 3 characters)
- **Phone number** — for SMS OTP verification (normalized to E.164 format)
- **Gender** — `male` or `female` (required for the matching algorithm)
- **Date of birth** — for age verification (18+) and matching preferences
- **Invite code** (if any) — records who invited you
- **Interested in** — `male`, `female`, `both` (optional)

### 2.2 Profile Data

When you create and edit your profile, we collect the following information:

- **Profile photo (avatar)** and **up to 6 additional photos** — uploaded from your device's gallery; stored under `avatars/{userId}/...` in Supabase Storage
- **Biography text (bio)**
- **City information** — you can either enter it manually or auto-detect via GPS with location permission (only the city name is stored; raw coordinates are not stored)
- **Interest tags (hashtags)**
- **Match preferences** — age range (min/max), interested gender
- **Social media links** (optional) — Instagram, Facebook, Snapchat usernames/links
- **Preferred languages** — the App stores `["tr"]` by default
- **Your invite code** — automatically generated by the App (8-character nanoid)

### 2.3 Location Data

- If you grant location permission, we read your device's approximate (Balanced accuracy) GPS coordinates via `expo-location` **solely to auto-populate the city field on your profile**.
- The coordinates are converted to a city name via reverse geocoding, and **only the city information** is written to our database. Raw latitude/longitude or precise location is not stored or shared.
- Location permission is optional; you can use the App by entering your city manually.
- We do not perform any background location tracking.

### 2.4 Device and Technical Data

- **Push notification token (Expo Push Token)** — for match and message notifications
- **Device information** (via `expo-device`, `expo-application`): device model, OS version, app version — used only for debugging and compatibility checks
- **IP address** — automatically transmitted with API requests; retained in short-term logs for rate limiting (abuse prevention) and security
- **Connection status** (`@react-native-community/netinfo`) — in device memory only, not sent to the server
- **JWT session token** — stored securely on your device via `expo-secure-store`

### 2.5 Activity and Game Data

- **League score** and **credit balance**
- **Credit transaction history** — match expenses, daily rewards, invite bonuses, game earnings
- **Daily login information** — last login date, consecutive day count
- **Match queue status** — whether you are in the queue
- **Match records** — users you matched with, match status, game score, message counters, like/pass decisions, abandonment info, rematch requests
- **Game session records** — start/end time, question count, total score

### 2.6 Communication Data (Anonymous Chat)

- **Chat messages** — profiles remain hidden until both parties send 20 messages each; messages are stored in the `messages` table with `(sender ID, match ID, content, timestamp)`
- Message content is **NOT** end-to-end encrypted; our server has the technical capability to read messages. This access may be required for abuse complaints and our legal obligations
- Messages are limited to a maximum of 1000 characters

### 2.7 Safety and Report Data

- **User reports** (`user_reports` table) — whom you reported, who reported, in which match
- **Karma score** — internal trust score based on reports and behavior
- **Feedback** (`feedback` table) — bug report/suggestion text you submit

### 2.8 Automatically Collected Data

- **Logs** — server errors, API request metadata (endpoint, status code, timestamp)
- **Cache data** (Redis) — OTP verification codes (5 min), game state (during the session), rate limiting counters — these are temporary and automatically deleted via TTL

## 7. Data Security

Measures we take to keep your data safe:

- **Encryption:** All client–server traffic is encrypted via HTTPS/TLS. Database connections are over SSL
- **Password security:** Passwords are hashed by Supabase Auth using a bcrypt-like one-way algorithm; no one, including us, can see the plain text
- **On-device:** JWT session tokens are encrypted on your device via `expo-secure-store` (iOS Keychain / Android Keystore)
- **Access control:** Row-Level Security (RLS) policies are active in the database; the API uses the service role only for authorized operations
- **Rate limiting:** IP/phone-based rate limiting on OTP, registration, and login endpoints
- **Authorization:** All protected endpoints go through JWT verification (Supabase JWT)
- **Helmet:** HTTP security headers are enabled

No system is 100% secure; in the event of a data breach, we will notify you and the competent authorities within a reasonable time as required by our legal obligations.
